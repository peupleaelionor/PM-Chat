# Packet Format

> All messages on the wire use the `MessageEnvelope` format. The server only sees encrypted payloads — it cannot read message content.

---

## MessageEnvelope Schema

```typescript
interface MessageEnvelope {
  version: number;           // Protocol version (currently 1)
  id: string;                // UUID v4 — unique message identifier
  conversationId: string;    // Target conversation
  senderId: string;          // Authenticated sender ID
  encryptedPayload: string;  // Base64-encoded AES-GCM ciphertext
  iv: string;                // Base64-encoded 12-byte initialization vector
  nonce: string;             // Base64-encoded 16-byte replay nonce
  timestamp: string;         // ISO 8601 creation timestamp
  expiresAt?: string;        // ISO 8601 expiry (optional)
  burnAfterReading?: boolean; // Self-destruct after read (optional)
  deliveredAt?: string;      // ISO 8601 delivery timestamp (server-set)
  readAt?: string;           // ISO 8601 read timestamp (server-set)
  attachmentId?: string;     // Reference to encrypted attachment (optional)
}
```

## Field Details

### `version` (required)

Protocol version number. Currently `1`. Used for forward compatibility — future clients can detect and handle older packet formats.

### `id` (required)

UUID v4 generated by the client. Uniquely identifies the message across the system.

### `encryptedPayload` (required)

The message content, encrypted with AES-GCM 256-bit using the shared ECDH-derived key.

**Encoding**: `plaintext → UTF-8 bytes → AES-GCM encrypt → base64 string`

The server stores this as an opaque blob. It cannot decrypt this field.

### `iv` (required)

12-byte initialization vector, randomly generated for each message.

**Why 12 bytes?** AES-GCM recommends 96-bit (12-byte) IVs for optimal performance and security. Each message gets a fresh random IV — never reuse IVs with the same key.

**Encoding**: `crypto.getRandomValues(12 bytes) → base64 string`

### `nonce` (required)

16-byte cryptographically random value used for replay protection.

**Server-side check**: `Redis SET NX nonce 1 EX 86400` — if the SET fails (key exists), the message is rejected as a replay.

**Encoding**: `crypto.getRandomValues(16 bytes) → base64 string`

### `timestamp` (required)

ISO 8601 timestamp of when the message was created on the client. The server may also record its own received timestamp.

### `expiresAt` (optional)

ISO 8601 timestamp after which the message should be auto-deleted. Used for self-destructing messages. MongoDB TTL indexes handle automatic cleanup.

### `burnAfterReading` (optional)

If `true`, the message should be deleted from the UI after the recipient reads it.

### `attachmentId` (optional)

Reference to a separately uploaded encrypted file attachment. Attachments are encrypted client-side with the same shared key before upload.

## Validation (Zod Schema)

```typescript
const MessageEnvelopeSchema = z.object({
  version: z.number().int().positive(),
  id: z.string().uuid(),
  conversationId: z.string().min(1),
  senderId: z.string().min(1),
  encryptedPayload: z.string().min(1),
  iv: z.string().min(1),
  nonce: z.string().min(1),
  timestamp: z.string().datetime(),
  expiresAt: z.string().datetime().optional(),
  burnAfterReading: z.boolean().optional(),
  deliveredAt: z.string().datetime().optional(),
  readAt: z.string().datetime().optional(),
  attachmentId: z.string().optional(),
});
```

All incoming messages are validated against this schema before processing. Invalid packets are rejected immediately.

---

## Compact Design Principles

1. **Minimal overhead**: Only essential fields are required
2. **Base64 encoding**: Binary data encoded as base64 for JSON transport compatibility
3. **No plaintext fields**: The server never sees unencrypted content
4. **Schema validation**: Every packet is validated on receipt
5. **Replay protection**: Nonce ensures each packet is unique
6. **Forward compatible**: Version field allows future protocol evolution
